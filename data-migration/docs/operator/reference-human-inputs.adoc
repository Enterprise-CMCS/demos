= Reference: Human inputs and review gates

The comprehensive list of every file a human edits, reviews, or signs
off before the data migration runs. It is grounded in the files the
pipeline actually consumes: each row names a real artifact and the code
that reads it. Use it as a pre-cutover checklist and as the index to the
deeper SME / developer pages.

Inputs fall into three tiers:

* *Tier A* -- files you _edit_; code consumes them directly.
* *Tier B* -- artifacts you _review and sign off_; their signature
  unblocks a gate.
* *Tier C* -- changes owned by the *DEMOS app team* (target schema),
  outside this repo, that the migration depends on.

== How to read this page

* *Owner* is who holds the pen: *Operator* (runs the cutover), *SME*
  (subject-matter expert confirming business mappings), or *DEMOS app
  team* (owns the Prisma-managed `demos_app` schema).
* *Phase / gate* maps to the pipeline stages in
  xref:reference-cli.adoc[the CLI reference] and the gate files in
  xref:reference-gates-state.adoc[gates and state].
* *Validation* is the fail-closed check or verifier that catches a
  missing or wrong input -- the forcing function that keeps a guess from
  silently passing.
* The <<status-snapshot,status snapshot>> at the bottom is dated prose,
  re-audited by hand; it is the only part of this page not enforced by
  `verify-doc-facts`.

The dominant Tier A pattern is the crosswalk lifecycle: a proposed CSV is
reviewed by an SME, transcribed into the live `04_crosswalks` SQL, and
guarded by a fail-closed `_check.sql` that stays red until the mapping is
complete.

[mermaid]
....
flowchart LR
  prop["proposed.csv"] --> rev["_review.md (SME)"]
  rev --> sql["04_crosswalks SQL"]
  sql --> chk["_check (fail-closed)"]
  chk --> gate["build / parity gate"]
....

== Tier A -- input files you edit

=== A.1 Code and enum crosswalks

Legacy MySQL reference codes mapped to DEMOS target values. Proposals
live under `reports/crosswalks/proposed/` (not read by code); confirmed
values are transcribed into `reports/crosswalks/<name>.csv`, loaded from
CSV by the crosswalks phase (the SQL files are DDL only), and enforced
by the matching `_check.sql`. The phase reads
`reports/crosswalks/registry.yaml` for the list of crosswalk tables to
load. Owner: *SME* confirms, *Operator* edits the CSV + registry.
Phase: `crosswalks` (Gate: crosswalk sign-off in
xref:../sme/howto-author-a-crosswalk-csv.adoc[the SME crosswalk how-to]).

(the file enumeration is). Re-audit before each rehearsal and at T-24h.
