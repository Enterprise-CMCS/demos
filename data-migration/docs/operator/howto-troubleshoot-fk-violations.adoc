= How-to: troubleshoot FK violations

`migrate constraints` re-validates every FK and **writes**
`state/fk_violations.csv` with any still-unvalidated FKs. If that list
is non-empty, the phase calls `die()` and the cutover halts. This is
how you diagnose.

== File format

`state/fk_violations.csv` columns (one row per invalid FK):

[cols="1,3"]
|===
| Column | Meaning

| `schema` | Schema of the table carrying the un-validated FK (`demos_app`).
| `table` | `demos_app.<table>` carrying the un-validated FK.
| `constraint_name` | The FK name (e.g. `fk_demonstration_application_id`).
|===

== Triage flow

[mermaid]
....
flowchart TD
  A[fk_violations.csv non-empty] --> B{< 30 min to fix?}
  B -- yes --> C[Fix upstream stg transform]
  C --> D[Re-run build + constraints]
  D --> E{empty now?}
  E -- yes --> F[Continue to parity]
  E -- no --> G[Rollback]
  B -- no --> G
....

== Common causes

. *Missing crosswalk row.* A reference value present in `mysql_raw`
  has no `legacy_int_cd -> demos_text_id` mapping. Fix in
  `reports/crosswalks/<rfrnc>.csv`; re-run `make crosswalks build`.
. *Stale id map.* The id map for the target table was populated
  before the source got new rows. Fix: re-run `make id_maps build`.
. *Pending/approved disagreement.* The pending row references a
  parent the approved flow has not materialised. Fix: review
  `reports/narrative/pending_approved_decisions.md` and adjust the rule for
  that pair.
. *Source data integrity error.* The source row legitimately points
  at a non-existent parent. Fix: open a parity warning and decide
  with the SME whether to drop the row or stub a placeholder parent.

== Re-running constraints alone

Once you have rebuilt the implicated stage:

[source,sh]
----
uv run migrate constraints
----

The phase is idempotent. It re-validates every FK and rewrites
`state/fk_violations.csv` with whatever remains un-validated.

== Escalate

If the violations cannot be resolved in 30 minutes during a real
cutover, *roll back*. See xref:howto-rollback.adoc[rollback].
