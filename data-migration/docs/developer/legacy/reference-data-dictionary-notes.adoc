= Legacy: DEMOS data dictionary -- hand-authored notes (superseded)
:toc: left

[CAUTION]
====
This is the **superseded** hand-authored data dictionary, preserved here
as an artifact for the per-column semantic *Notes* that the generated
page does not carry. The live page is
xref:../reference-data-dictionary.adoc[Reference: DEMOS data dictionary],
whose structural facts (columns, types, nullability, defaults, PKs, FKs,
unique indexes, CHECK constraints, history shadows) are generated from the
pinned Prisma DDL by `docs/tools/data_dictionary_to_adoc.py` and cannot
drift from the schema. Do not edit this file expecting the live page to
change; it is not included in the built wiki.
====

A column-level reference for every table in the new DEMOS PostgreSQL
database. The authoritative source is the Prisma schema in the DEMOS
app repo, materialised as the SQL artifact `migrate ddl` fetches and
pins via `reports/prisma_ddl.sha256`; the `*_history` tables are part
of that same Prisma artifact (DEMOS audit shape), not authored by this
repo. This page collates that DDL into one place with the semantic
notes that the schema sources leave implicit.

== Scope

* The primary tables and their `*_history` audit tables in the
  `demos_app` schema (all Prisma-owned, per
  xref:../shared/schema-overview.adoc[the schema overview]).
* Static-constraint and type-limiter tables are summarised here and
  linked through to xref:../sme/reference-static-constraint-tables.adoc[the
  SME-facing static-constraint reference] for the full DDL.
* Foreign keys are listed in *expected target* form even though they
  ship in the Prisma artifact and are re-applied late by `migrate
  constraints` (see xref:explanation-fk-strategy.adoc[FK strategy]).
* For the column-level join *back to the MySQL PMDA source*, see
  xref:reference-source-target-columns.adoc[the source/target column
  map].

== Conventions

[cols="1,4"]
|===
| Convention | What it means

| `uuid` primary keys
| Every event-bearing entity (`application`, `demonstration`,
  `deliverable`, `document`, ...) keys on `id uuid`. UUIDs are
  generated by `gen_random_uuid()` and are stable across rebuilds
  because they flow through `migration._id_map_*` (see
  xref:reference-id-maps.adoc[id maps]).

| `text` primary keys
| Static-constraint and type-limiter tables key on `id text` (e.g.
  `'pending'`, `'approved'`). See
  xref:../sme/reference-static-constraint-tables.adoc[static-constraint
  tables] for why.

| `timestamptz` everywhere
| `created_at`, `updated_at`, `effective_date`, `expiration_date`,
  `due_date` are `timestamptz NOT NULL DEFAULT now()` on the parent.
  The `*_history` audit tables instead carry `modified_at timestamptz
  NOT NULL DEFAULT CURRENT_TIMESTAMP`.

| `_history` audit tables
| Every parent with history has a matching Prisma-owned
  `<table>_history` table with `revision_id SERIAL PRIMARY KEY`,
  `revision_type revision_type_enum ('I','U','D')`, `modified_at
  timestamptz`, then the parent columns (mirroring parent nullability).
  This migration leaves these tables empty; DEMOS populates them via its
  capture triggers post-cutover. See
  xref:explanation-history-backfill.adoc[history is DEMOS-owned].

| JSONB columns
| `validation_data`, `response`, and `token_list` are `jsonb NOT NULL`
  (there is no `event_data` column -- the Prisma schema has no `event`
  table). DEMOS owns these live columns and validates them at runtime,
  so this migration wires no CONSTRAINT TRIGGER on any of them. The
  only JSONB shape trigger is the budget-neutrality parity oracle on
  the migration-private `migration.bn_workbook_detail`; see
  xref:reference-jsonb-schema-registry.adoc[JSONB schema registry].

| Composite uniqueness for FK chaining
| Many tables ship redundant composite unique keys
  (e.g. `UNIQUE (id, application_type_id)`) so that downstream tables
  can FK into the *combined* parent identity. This is the mechanism
  the application uses to keep status / type pairs consistent across
  rows.
|===

[NOTE]
====
Column tables below use the same header in every section:
`Column | Type | Null | Default / Key | Notes`.

* *Null* is `N` for `NOT NULL`, `Y` otherwise.
* *Default / Key* lists `PK`, `UK`, FK target, and any `DEFAULT`.
* History columns are not repeated; for any parent below, its
  `_history` audit table has the columns shown plus the three
  audit-only columns (`revision_id`, `revision_type`, `modified_at`),
  with the parent columns mirroring the parent's nullability.
====

== Static-constraint and type-limiter tables

The 29 static-constraint tables and 14 type-limiter tables are
single-column `id text PRIMARY KEY` lookups (with a few that carry
extra boolean attributes, e.g. `demos_app.deliverable_action_type`). The
canonical DDL ships in the Prisma artifact alongside the rest of
`demos_app.*`; this repo only seeds their data via
`sql/02_seeds_static/` and `sql/03_seeds_limiters/`.

For the full DDL and seed values, see
xref:../sme/reference-static-constraint-tables.adoc[the SME-facing
static-constraint reference]. For the rationale behind text IDs, see
xref:../sme/explanation-data-shape-decisions.adoc[data-shape
decisions].

== History table contract

The `*_history` tables are *Prisma-owned* (DEMOS baseline migration),
not authored by this repo. They share the DEMOS audit shape:

[source,sql]
----
CREATE TABLE demos_app.<table>_history (
  revision_id   SERIAL PRIMARY KEY,
  revision_type revision_type_enum NOT NULL,  -- 'I' | 'U' | 'D'
  modified_at   timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP,
  -- the parent columns, mirroring demos_app.<table> nullability
  ...
);
----

The `log_changes_*` capture triggers that populate these on `INSERT`,
`UPDATE`, and `DELETE` are also DEMOS-owned
(`server/src/sql/history_triggers.sql`, applied via
`refreshDbObjects.ts`) -- `sql/32_app_triggers/` in this repo is
intentionally empty. This migration does not populate the history
tables; they ship empty and the DEMOS capture triggers fill them from
cutover forward (see
xref:explanation-history-backfill.adoc[history is DEMOS-owned]).
