= Canonical migration specification: MySQL -> DEMOS PostgreSQL warm cutover
:diagram-nocache: true

Solo-operator migration spec for the CMS Medicaid Demonstrations management system.
Source: legacy MySQL (no FKs, ~300 tables, <50 GB).
Target: DEMOS PostgreSQL (heavy remodel, FKs, JSONB, history tables, triggers).
Strategy: warm cutover into a brand-new application; old MySQL goes read-only and users flip URLs.

This page is the unified plan; the audience books in
xref:../operator/index.adoc[operator],
xref:../developer/index.adoc[developer], and
xref:../sme/index.adoc[SME] reorganise its content for specific
readers and link back here for the full picture.

== 1. Inputs recap

* *Source DB:* MySQL CMS Medicaid Demonstrations, ~300 tables, no FKs, <50 GB.
* *Target DB:* DEMOS PostgreSQL schema (provided ER model). Heavy remodel into `application` / `demonstration` / `amendment` / `extension` / `deliverable` core, tag system, JSONB consolidations, FKs / CHECKs / triggers, and history tables.
* *Application:* brand-new app on DEMOS. Old app goes read-only at cutover; users flip URLs.
* *Tooling:* `pgloader` for raw MySQL \-> PG load, plain numbered SQL scripts for transforms (no dbt), DuckDB sidecar for offline analysis and the non-gating load-fidelity row-count checks (parity itself is pure-Postgres).
* *FK derivation:* inferred from naming conventions in the supplied digraph, then team-reviewed.
* *Operator profile:* solo, minimal dbt experience.
* *Downtime tolerance:* up to a week; planned freeze targeted at hours, not days.

== 5. Why this changes the build plan

Versus a generic MySQL \-> PG migration:

. *History is DEMOS-owned, not migrated.* Per the SMEs, the Prisma-owned DEMOS `*_history` tables do not have to be populated at cutover; they are left empty and the DEMOS `log_changes_*` capture triggers fill them post-cutover. The migration does not backfill from the MySQL `*_hstry` tables.
. *Reference-code crosswalks become a first-class artifact.* Every `*_rfrnc` int \-> DEMOS text-id mapping must be authored as CSV, reviewed, and seeded into `mysql_raw.crosswalk_*` tables before `stg` builds.
. *Program-detail consolidation is templated.* ~40 near-identical source tables get one templated SQL transform (one file) parameterized by table name + tag-name; not 40 separate hand-written transforms.
. *Pending/approved unification is its own workstream.* For each pending+approved entity pair, decide:
 ** same business object across both \-> single `application` row, status flow captures the lifecycle;
 ** separate transient draft \-> only the most recent pending row materializes if nothing in approved;
 ** inconsistent (pending and approved disagree) \-> approved wins; pending logged as a parity warning.
. *ID mapping is unavoidable.* MySQL `int` PKs \-> DEMOS `uuid` PKs; we need `migration._id_map_<table>` for every data table to reconstruct FKs.

== 6. Cutover state machine

[mermaid]
....
stateDiagram-v2
  [*] --> Live
  Live --> Freeze: planned start
  Freeze --> Delta: pgloader delta
  Delta --> Build: stg + app + history
  Build --> Constraints: FKs + triggers + indexes
  Constraints --> Parity: pure-PG checks
  Parity --> Flip: go decision
  Parity --> Rollback: no-go
  Flip --> Hypercare
  Hypercare --> Decom: stable for N days
  Rollback --> Live
  Decom --> [*]
....

*Legend.*
`Live` = both DBs writable, old app serving traffic.
`Freeze` = MySQL writes paused; old app set read-only.
`Delta` = pgloader pulls last delta into `mysql_raw`.
`Build` = numbered SQL rebuilds `stg` and `demos_app`. The `demos_app.*_history` tables are left empty for DEMOS to populate post-cutover (this migration does not backfill history).
`Constraints` = FKs validated; triggers attached; indexes built.
`Parity` = pure-Postgres checks (row counts, numeric sums, JSONB shape, pending/approved, reconstructed FK orphans, demonstration ID provenance, users/person integrity, demonstration completeness, RBAC integrity for the system-role / person-state / demonstration-role assignments, active-users coverage, plus the non-gating Approved-held and scope-coverage logs -- 31 in all) comparing `mysql_raw` and `demos_app` in one PG cluster. Not a DuckDB job.
`Flip` = go-live verification. PMDA (legacy) and DEMOS (new) are separate apps on separate URLs; the dev team owns the URL/redirect work outside this repo. This phase confirms DEMOS healthz is responsive and that PMDA has been placed in read-only mode.
`Hypercare` = on-call window for tickets; no schema changes.
`Decom` = MySQL reader retired.
`Rollback` = restore old app to read-write; reset to `Live`; root cause + reschedule.

=== Why Python over bash

The orchestration started in bash and was ported to Python early in Week 1 because:

. *Solo-operator readability.* Coming back to a Typer subcommand in Week 7 (UAT + perf bug-fix rounds) is faster to re-understand than a heredoc-laden bash script.
. *Type-checking + tests.* `ty check` and `pytest` catch bugs that bash silently passes. A 1-day port pays back in Weeks 6-9.
. *Robust template rendering.* Jinja2 with `StrictUndefined` (in `lib.render_template`) handles arbitrary URLs that contain `|`, `&`, `/`, refuses to silently ship a half-rendered file when a marker is missing, and emits rendered files as `0600` so embedded credentials are not world-readable; `sed` does not handle any of this without extensive escaping.
. *Proper CSV/JSON handling.* `csv.DictReader`, `pyyaml`, and `jsonschema` are stdlib-or-near. The bash equivalents are fragile.
. *One language end-to-end.* Parity (pure Postgres), FK validation loops, JSONB schema checks, manual-override YAML overlays -- all fit naturally in Python and would be painful in bash.

Bash remains as a one-line shim (`scripts/cutover.sh`) so the muscle-memory invocation still works.

=== Gate state mechanics

`migration/lib.py` exposes `mark_gate(name)`, `require_gate(name)`, `clear_gate(name)`, and `list_gates()`. Each `mark_gate("foo")` writes `state/foo.ok` containing the UTC timestamp. `require_gate("foo")` calls `die()` (which ``sys.exit(1)``s) if the file is absent. `migrate status` renders the gate table in the terminal via `rich.table` on `stdout_console`; operator-facing logs go to `stderr_console` so `migrate status` output is pipeable. Re-running an already-completed phase is allowed and idempotent: it overwrites `.ok` with a fresh timestamp.

Two helpers reduce boilerplate inside phase modules:

* `+lib.phase(name, requires=..., mark=True)+` is a decorator that wraps a phase function with `require_gate(<requires>)` on entry and `mark_gate(<name>)` on clean exit. `mark=False` lets phases like `parity` decide for themselves when to mark (only on GREEN or `--accept-pending`).
* `lib.confirm(prompt, expected="y")` reads stdin and dies if `MIGRATE_NONINTERACTIVE=1` is set -- the operator gets a loud refusal rather than every gated phase silently aborting in CI / piped invocations.

The canonical phase order (`lib.PHASES`) -- used by `migrate status` and `migrate resume` -- is: `preflight, freeze, delta, build_stg, build_app, constraints, parity, flip, smoke, decom`. `migrate resume` walks this order, skipping phases whose `.ok` already exists; `decom` is deliberately excluded from `resume` (`cli._RESUME_PHASES`) as a Day-7+ chore outside the cutover-day sequence.

== 13. Risks and mitigations

* *Pending/approved data inconsistencies.* Mitigation: explicit decision doc per pair; parity check that every pending row is accounted for; warnings logged.
* *Crosswalk completeness.* Mitigation: every `*_rfrnc` table in scope has a CSV; SQL check confirms every distinct legacy code in `mysql_raw` has a mapping.
* *Program-detail tag explosion.* Mitigation: tag dictionary signed off in Week 2; templated transform reviewed once vs. 40 hand-written.
* *Document linkage gaps.* Doc tables reference many parent entities; some may not migrate cleanly. Mitigation: orphan-doc parity report; doc `description` field captures source for forensics.
* *Backup tables hiding live data.* Mitigation: pre-load comparison of `*_bkup` vs base to confirm bkup is stale; documented in drop list.
* *JSONB BN shape drift over time.* Mitigation: JSON Schema validation as parity check; representative sample manually reviewed by SME.
* *Solo execution under pressure.* Mitigation: two timed rehearsals, scripted go/no-go, pre-written comms, backup operator.
* *Reference-table semantics changing post-freeze.* Mitigation: crosswalks frozen at Week 2; any new reference codes added during cutover week trigger an explicit review gate.
* *New-app behavior diverging from old-app expectations.* Mitigation: UAT in Week 7 with real users on cloned data; bug list fed back into transform fixes.
