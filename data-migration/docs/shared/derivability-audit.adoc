*Question.* For every column in the DEMOS (target) tables that does *not* come
1:1 from the mapped MySQL (source) table, can it be _derived_ from other
("Nth") source tables -- the way `person.person_type_id` was derived in
`8bf725d` (`users` -> `user_role_asgnmt` -> `crosswalk_role_person_type`)?

*Answer.* *No -- not entirely.* Most non-copy target columns are derivable (via
id-mint, crosswalk, tag-pivot, or a join to an Nth table), but a hard residue
is not derivable from any source table and must be supplied by a constant, a
target-schema concession, or a human. Of the {derivability-gap-in-scope}
currently in-scope target gaps, {derivability-derivable} are derivable and
{derivability-nonderivable} are not (`users.cognito_subject`,
`private_comment.deliverable_id`, `public_comment.deliverable_id`). See
<<in-scope-gap-verdict>>.

[NOTE]
====
*Coverage caveat.* This is only fully _answerable_ for the
{derivability-mapped-tables} target tables mapped today;
{derivability-pending-tables} of {derivability-live-tables} live target tables
have no mapping rows yet ({derivability-gap-pending} NOT-NULL columns,
`GAP_table_pending`). That raw count overstates the work: the reproducible
bucket split (<<unmapped-table-buckets>>) shows only the
{derivability-foldable-tables} `foldable` tables need a *source-derived*
mapping -- {derivability-gap-foldable} NOT-NULL columns. The rest are `seeded`
(static vocabularies loaded from reference data) or `net_new` (no source at
all), so they are out of scope for derivation, not blocked on it.
====

== File-by-file review

=== `schema_snapshot/` -- source structure

The standout is `foreign_keys.csv` = 0 rows: the source declares *no* FKs, so
derivation join paths cannot be read from declared keys. They are reconstructed
by `fk_candidates.csv` + `fk_overrides.yaml`. *Every Nth-table derivation is
only as trustworthy as those inferred joins.* `columns.csv` (every column of
all source tables) is the map of what an Nth-table join _can_ supply;
`table_stats.csv` says which Nth tables are populated; `indexes.csv` says
whether an Nth join is 1:1 or fan-out; `triggers.csv` is source business logic
that may need re-derivation.

=== `reference_data/` -- source data

* *68 `*_rfrnc` lookup tables* -- uniform shape `<x>_cd, <x>_name,
  [dsply_sqnc_num], [dltd_ind], creatd_dt`. These are the legacy *code
  vocabularies* every `crosswalk:*` derivation translates into DEMOS text IDs.
* *5 `v_*` views* -- denormalized demonstration-type rows feeding the
  `tag_pivot` derivations and the status overlay.
* A few lookups encode their own join path (e.g. `email_subj_rfrnc` ->
  `email_tmplt_cd`, `geo_ansi_state_rfrnc` -> `rgnl_ofc_cd`).

== Bottom line

Within the mapped scope, the existing transform vocabulary (`id_map`,
`crosswalk:*`, `tag_pivot:*`, `derive`) covers all but
{derivability-nonderivable} target NOT-NULL columns, and those are structurally
underivable from the source -- already handled by a schema concession
(`cognito_subject`) or still open (`*_comment.deliverable_id`). The larger risk
is *coverage, not derivability* -- but narrowly scoped: of the
{derivability-pending-tables} unmapped tables, only the
{derivability-foldable-tables} `foldable` ones ({derivability-gap-foldable}
NOT-NULL columns) need a source-derived mapping (<<unmapped-table-buckets>>);
the rest are `seeded` or `net_new`. The two non-derivable patterns above will
recur in that foldable residue.
