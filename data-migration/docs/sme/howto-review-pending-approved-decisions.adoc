= How-to: review pending/approved decisions

Every MySQL `mdcd_<x>` / `mdcd_pendg_<x>` pair has a rule for how the
two tables collapse into the single DEMOS `application` flow. The
SME review is the source of truth for these rules.

== Where the file lives

`reports/narrative/pending_approved_decisions.md`. Live contents:
xref:reference-pending-approved-rules.adoc[reference page].

== The default rule

Collapse to a single `application` row keyed by demonstration
identity; status flow captures the lifecycle.

* `mdcd_pendg_*` rows that have *no* approved counterpart become
  `application` with status `'pending'`.
* `mdcd_pendg_*` rows that *do* have an approved counterpart fold
  into the approved demonstration's history.
* When the two disagree, *approved wins*. The pending difference is
  logged as a parity warning.

== When to deviate

. *Separate transient draft.* The pending row is genuinely separate
  from any approved row -- e.g. a different draft amendment. Rule:
  the pending row materializes only when nothing in approved
  references the same identity.
. *Composite identity.* Pending and approved keys do not align
  trivially; identity is composed of multiple columns. Rule: collapse
  on the composite, document the columns in `notes`.
. *Source-side truncation.* Pending rows expire after N days in
  the source app and are deleted; the rule should handle "missing"
  pending gracefully.

== Review checklist for each row

* [ ] Pair is named in source-table form (e.g. `mdcd_demo_amndmt /
      mdcd_pendg_demo_amndmt`).
* [ ] DEMOS target is named.
* [ ] Rule is one of: collapse / 1:1 / drop / parity-warn.
* [ ] Tie-break is named (almost always `approved wins`).
* [ ] Notes capture any composite-key or schema oddity.

== Authoring

The file is markdown. Add a row, run `make build` end-to-end on a
rehearsal, and confirm the parity report does not flag any pending
rows as unaccounted-for. If it does, refine the rule.

== Sign-off

The bottom of the file has a sign-off block. Fill in your name and
date when you are confident the rules cover every observed pair.
